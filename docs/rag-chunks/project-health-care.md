# Health Care 프로젝트 (ID: 4)

## 개요
- 제목: AI 기반 식단 분석 시스템 (HealCare)
- 목표: AI 캐릭터(페르소나) 중심의 대화형 다이어트/건강 메신저 제작. YOLO 이미지 분석과 사용자 건강 프로필 결합.
- 기반 스택: Django 5.2, YOLOv9, Gemini 2.0 Flash, AWS RDS MySQL.
- 프론트엔드: HTML/CSS/JS (Bootstrap + jQuery 활용하여 카카오톡 형태 UI, AJAX 통신 적용).

## 핵심 기능 상세 (STRUCTURE / FLOW)
1. **AI 페르소나 시스템**: 
   - 김계진(열정 트레이너 밈), 박민아(친한 친구 음슴체), 윤서린(전문적 비서). 프롬프트 디렉토리(`gemini/prompts/`)에서 텍스트 기반 캐릭터 파일을 실시간 로딩하여 프롬프트로 병합. UI 좌측 아이콘 클릭 시 즉시 컨텍스트 리셋 및 새 모델로 변환(채팅창 비움 효과).
2. **음식 이미지 분석 (YOLOv9 Custom/COCO 병행)**:
   - Python 런타임에서 두 개의 PT 모델(`yolov9s.pt(COCO, conf:0.6)` / `best.pt(Custom, conf:0.9)`) 병렬 predict 추론 스크립트 실행. 
   - 파싱 결과를 세트로 묶음 후 중복 제거, 최종 DB(Calorie 테이블) 파라미터 매칭(칼로리, 탄탄지 조회) 후 결과 프롬프트에 담아 전송.
3. **사용자 개인정보 반영 (PersonalData)**:
   - AWS RDS에 보관된 사용자 키, 몸무게, 알러지를 프롬프트에 문자열로 주입(`full_prompt`). "알레르기가 있는 나에게 조언해줘" 동작을 체계적으로 백그라운드에서 실시간 통합 수행.
4. **접근 제어 (Interceptor/JS Redirection)**:
   - 세션 ID 부재 시 비회원 메인 페이지 강제 이동 (JavaScript `window.location` + jQuery AJAX 백엔드 세션 체크).

## 트러블 슈팅 (TROUBLE SHOOTING)
1. **커스텀 음식 모델 정확도 이슈**:
   - 증상: 삼겹살 등 일부 데이터가 다른 음식으로 판독(65%).
   - 조치: 모델 분리 병행. 글로벌 음식과 광범위 객체는 COCO의 `0.6` conf로 허용, 반면 커스텀(한국요리 60여종)은 정확성을 높이기 위해 `0.9` conf로 올리고 중복을 교집합 제거. 데이터 재학습 병행(삼겹살 95% 이상으로 상향).
2. **Gemini 2.0 Flash 기억력 한계 방어 (토큰 압축)**:
   - 한계: 턴수 제한 및 장기 대화의 구 버퍼로 인해 AI가 과거 맥락 소실 및 토큰 비용 폭발.
   - 해결 방안: 파일 기반 대화 로그 시스템 마련. `chatlog_{userId}` 파일을 백엔드에 기록하고, 통상 **6턴의 요청**마다 카운터를 갱신하여 턴이 만료되면 대화 파일 삭제/초기화 시킴. 따라서 가장 핵심적인 대화 맥락만 계속 프롬프트에 제공하며 약 70% 비용과 속도를 최적화.
3. **이미지 중복 저장 문제 막기**:
   - 증상: 사용자가 매 턴 이미지를 올리면 서버/AWS 하드 용량이 막대하게 소모.
   - 대처: 업로드 로직에 가드 `deleteImage` 함수 구축. 동일 세션(유저)이 새 이미지를 올릴 시 직전에 저장된 객체의 디렉토리 실데이터 `os.remove` 수행 ➔ 무조건 최신 음식 1개만 기억하게 강제.
