# Session Sharing Performance 프로젝트 상세 문서

## 프로젝트 개요
- **프로젝트명**: Session Sharing Performance (세션 저장소 방식에 따른 응답 성능 교차 검증)
- **개발 기간**: 2026년
- **주요 목적**: 다중 WAS 아키텍처 환경에서 세션 공유를 위해 대체로 거론되는 세 가지 저장소(Redis, Session Clustering, RDBMS) 모델을 비교 분석합니다. 각 방식이 저사양부터 고성능에 이르는 하드웨어 환경에서 어떤 실제적 성능 차이를 보이는지, 상세한 부하 테스트와 시각화를 통해 검증했습니다.

## 1. 테스트 목적 및 가설 수립

부하 테스트를 시작하기 앞서, 세션 저장소 성능에 대한 명확한 검증을 위해 다음과 같은 세 가지 핵심 가설(추측)을 수립했습니다.

1. **In-Memory의 우위?** 인메모리 방식인 Redis가 물리적으로 가장 빠른 속도를 보장할 것이다.
2. **RAM 용량의 변수?** In-Memory 방식은 메모리 의존도가 높으므로, RAM이 상대적으로 낮은 환경에선 병목과 속도 저하가 클 것이다.
3. **읽기/쓰기 수준에서의 차이?** 단순 텍스트 수준의 세션을 읽고 쓰는 작업이라면 굳이 Redis를 쓰지 않고 RDBMS를 사용하더라도 큰 성능 차이는 보이지 않을 수 있을 것이다.

## 2. 테스트 구현 및 세팅 환경

각 방식이 단순한 이론으로 끝나지 않도록 철저히 통제된 5단계의 테스트 세팅을 설계 및 구현했습니다.

### 세팅 1: 데이터 크기 제어 및 API 구성
하드웨어 및 저장소 간 확실한 갭(Gap)을 보기 위해 일반적인 세션보다 무거운 **1KB 크기의 더미 데이터**를 구성하여 세션에 할당했습니다.
- **Write 작업 (`POST /api/login`)**: 1KB 데이터가 적재된 User 정보를 세션에 생성합니다.
- **Read 작업 (`GET /api/user`)**: 생성된 세션을 반복적으로 읽어 반환합니다.

### 세팅 2: Hazelcast 클러스터링 구성 방식
Spring Session Clustering을 구현하기 위해 노드 간 세션 복제가 가능하도록 **Hazelcast** 설정을 구성했습니다.
효율성을 위해 브로드캐스트 기반의 멀티캐스트 방식을 사용하지 않고, TCP/IP 네트워크를 이용해 서로 노드를 직접 명시(발견)하도록 구현했습니다.

### 세팅 3: Docker를 통한 컨테이너 및 이중화 포트 분리
아키텍처의 일관성을 맞추기 위해 모든 프로젝트와 DB는 로드 밸런싱 상황을 모사한 Docker Compose 환경에 실행됩니다. 읽기와 쓰기 테스트를 위해 각 방식별로 2개의 WAS 노드로 분리합니다.
- Redis 노드 (7070, 7071)
- RDBMS/MySQL 노드 (8080, 8081)
- Hazelcast Clustering 노드 (9090, 9091)

### 세팅 4: JMeter 부하 시나리오 및 통제 로직
일정한 조건으로 과격한 동시 접속 발생을 유도했습니다.
- **트래픽 조건**: 1,000명의 가상 사용자가 20초간 지속하여 요청을 보내는 패턴을 총 100회씩 반복했습니다.
- **간섭 억제**: 특정 저장소 테스트로 누적된 하드웨어 피로 및 캐시 찌꺼기가 남는 것을 막기 위해, 테스트들 사이에 **15분의 긴 쿨타임(Delay)**을 엄격하게 적용해 결과를 도출합니다.

### 세팅 5: 3단계 하드웨어 별 교차 테스트 
사양별 하드웨어 특성에 따른 변수를 확인하기 위해 세 가지 다른 물리 기반 환경을 구축하여 동일하게 비교했습니다.
- **Desktop**: 고성능 PC 환경 (10-Core, DDR4 16GB RAM)
- **Laptop**: 중간 사양 컴퓨팅 (6-Core, DDR4 32GB RAM 등)
- **Raspberry Pi**: 저사양 임베디드 장비 (4-Core, DDR4 8GB RAM)

## 3. 핵심 결과 도출 및 교차 분석

Python을 활용한 CSV 응답 시간 궤적을 확인했을 때 예상된 가설들과는 상당히 다른 의미 있는 결과들이 도출되었습니다.

### Redis vs Clustering 성능이 큰 차이가 없던 이유?
결과적으로 둘은 거의 비슷하고 어떤 상황에선 Clustering(Hazelcast)이 Read 속도에서 Redis를 앞서기도 했습니다.
- **원인 분석**: 도입된 Session Clustering 방식 역시 In-Memory를 기반으로 합니다. 게다가 Redis 방식은 매번 외부 노드에 접근하기 위한 **네트워크 IO 비용**이 수반되는 반면, Hazelcast Clustering은 다른 WAS의 세션 데이터를 스스로의 노드 내부 메모리에 이미 동기화(복사)해 두기 때문에 Read IO 과정이 훨씬 단축되는 강점을 보였습니다.

### MySQL(RDBMS)의 결과가 유독 처참했던 이유?
저사양 기기(라즈베리파이 등)에서 조회 성능이 최대 수 만 ms 에 다다르며 극심한 병목과 실패를 겪었습니다.
- **원인 분석 1 (태생적 차이)**: 하드디스크 중심의 On-Disk 저장소의 물리적 IO 속도는 In-Memory를 극복하기 어려웠습니다.
- **원인 분석 2 (Context Switching)**: Redis는 단순한 싱글 스레드 이벤트 루프 모델로 동작해서 스레드 전환 비용(Context Switching)이 없지만, RDBMS 방식은 MySQL Connection Pool 분배 과정에서 많은 비용을 낭비했습니다.
- **원인 분석 3 (USB 대역폭)**: 특히 라즈베리파이는 내부 저장소가 아닌 외부 USB를 보조 장치로 장착한 환경이었으므로 RDBMS의 쓰기/읽기 지연(Disk IOps)이 압도적으로 터질 수밖에 없었습니다.

## 4. 트러블 슈팅 내역 (테스트 절망편)

부하 테스트를 진행하면서 발생했던 겪었던 시스템 및 인프라상 치명적 오류 극복 경험입니다.

### 1. 테스트 절망편 1: 극심한 메모리 부족 (OOM) 현상
- **문제**: 부하를 주기 시작한 직후 메모리 리소스가 터지며 서버 프로세스들이 Out Of Memory로 함께 셧다운되는 증상을 다수 경험했습니다.
- **대응 방향**: 애플리케이션 프로세스 수준에서의 저지연 GC(Garbage Collector) 튜닝과, 도커 내 리소스 제약을 면밀히 파악하면서 부하량을 분산할 필요성을 체감했습니다.

### 2. 테스트 절망편 2: Connection Pool Size 초기화 한계 (Pool Size = 10)
- **문제**: RDBMS 방식 테스트 중 어느 순간 병목 현상이 폭증하여 뻗는 현상이 발생했습니다.
- **해결 원인**: JDBC 커넥션 풀 사이즈를 기본값인 10으로 방치한 것이 문제였습니다. 병렬로 밀려드는 1,000 유저 스레드의 요청을 풀 카운트 10개가 돌려 막다 보니 DB I/O보다 먼저 WAS 앞단에서 막혀버리는(Block) 현상이었음을 파악하고, 커넥션 튜닝 후 부하를 감당해 내었습니다.
